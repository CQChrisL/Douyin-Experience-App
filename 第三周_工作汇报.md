# 第三周汇报文档

## 1. 本周总体进度总结
第三周通过架构重构与底层性能深度优化，主要完成**Demo 的功能丰富和性能优化**，主要完成了以下 4 个任务：
1. **多 Tab 架构重构**：引入 `ViewPager2` + `Fragment`，将原来的单页应用改造为“推荐、经验、同城”三个栏目，实现业务解耦。
2. **真实远端网络接入**：废弃本地 mock 数据，构建了全局单例的 `NetworkManager`，通过 OkHttp 异步向 `picsum.photos` 真实接口拉取 JSON 并解析。
3. **开机预加载**：实现了类抖音的全屏沉浸式 UI（Edge-to-Edge），并通过单页面 Overlay 方案，将“开机动画”与“后台拉网”并行执行，实现了“秒开”体验。
4. **底层引擎换核与缓存击穿优化**：通过自定义 `AppGlideModule`，将 Glide 的底层网络加载器替换为共享连接池的 `OkHttpClient`；同时在页面跳转时尝试修复内存缓存未命中的问题。

## 2. 学习路径与核心思考逻辑
在整个重构过程中，主要通过**架构解耦与底层计算机系统级资源调度的深入分析**来理解架构：
- **Fragment 与 ViewPager2 的底层生命周期**：理解了 Fragment 作为轻量级视图控制器的强大之处。体会到 ViewPager2 底层继承自 RecyclerView，从而自动拥有极其霸道的视图复用和越界回收销毁（`onDestroyView`）能力，极大降低了复杂多 Tab 应用的内存消耗。
- **OkHttpClient 资源池的全局复用**：认识到在 App 生命周期内频繁实例化网络客户端会引发连接池冗余与线程爆炸。通过全局单例化，使得图片拉取与 JSON 业务请求完美共享底层的 TCP `ConnectionPool` 和线程 `Dispatcher`。
- **配置隔离与浅拷贝思想**：学习了通过 `newBuilder()` 对对象进行浅拷贝：使得图片媒体层既能享有独立的更长超时配置，又能免受业务层 Token 拦截器的污染。
- **多线程与事件流向**：理解了在 `onCreate` 中，主线程通过SplashOverlay实现开屏动画，同时子线程通过 `enqueue` 同步执行 I/O 拉取；最后通过 `Handler` 进行跨线程的 UI 渲染汇报，实现了极其精妙的多线程协作并发模型，提升用户体验。

## 3. 遇到的疑难杂症及解决方案
1. **共享元素转场导致的详情页卡顿和主线程假死**
   - **问题**：在 `DetailActivity` 中，原先利用 `supportPostponeEnterTransition()` 冻结转场等待图片加载完成。但由于列表页缩略图使用了 `.centerCrop()` 而详情页未使用，导致 Glide 生成了不同的缓存 Key。这引发了内存缓存 Miss，Glide 被迫去本地磁盘或网络重新解码，由于设备 CPU 和 I/O 限制，导致主线程死锁长达 2~3 秒，引发严重卡顿与黑屏。
   - **解决**：采取了“降级与预判”结合的思路：移除可能导致主线程假死的冻结等待，转而在 `DetailActivity` 的加载链中注入 `.thumbnail(Glide.with(this).load(url).centerCrop())`。使得引擎瞬间强制命中列表页刚刚留在内存中的缩略图，实现了“0 毫秒”无缝起播，随后原图在后台平滑过渡。但目前仍然有卡顿问题，目前不知道问题在哪。

2. **Android 沉浸式状态栏（Edge-to-Edge）导致的顶部视图重叠**
   - **问题**：在通过透明状态栏实现类似抖音的全屏沉浸式观感时，顶部的 `TabLayout` 文字导航会被系统自带的时间、电量等状态栏图标直接物理遮挡，导致难以辨认。
   - **解决**：通过调用 `WindowCompat.setDecorFitsSystemWindows(window, false)` 开启全屏绘制延伸，并在代码中利用 `WindowInsetsCompat` 动态获取当前设备的真实状态栏高度。最终将高度精准地作为 padding 设置给顶部导航栏，在保持背景沉浸的同时，实现了完美的视觉防遮挡避让。
