# 第二周汇报文档

## 1. 本周总体进度总结
第二周的总体上实现了工业级框架引入与高级交互设计。主要完成了以下 3 个阶段的架构重构与体验优化：
1. **图片加载架构重构**：废弃了第一周手写的 `ImageLoader`，引入了 Android 业界标杆框架 **Glide**，接管所有的网络调度与多级缓存。
2. **共享元素转场动画 (Shared Element Transition)**：抛弃了传统的页面左右推入效果，实现了从列表页的小卡片点击后，图片平滑“飞入”放大到详情页的视觉动效。
3. **交互与状态拓展**：
   - 引入官方 `SwipeRefreshLayout` 实现了瀑布流的下拉刷新逻辑。
   - 在详情页引入了“点赞状态机”及对应的 UI 动态反馈机制。

## 2. 学习路径与核心思考逻辑
本周的学习使我对客户端性能与内存管理有了更深层次的认识：
- **资源绝对稀缺性**：移动端更重要的资源是内存和磁盘缓存，所以为了节省手机空间资源，系统通过极端的 `LRU` 甚至直接 `Kill Process` 来保证存活，因此废弃旧资源是常态。
- **Glide 的降维打击**：理解了其“活动资源 (Active Resources) -> 内存 (LruCache) -> 磁盘 (DiskLruCache) -> 网络”的四级架构，以及它如何通过注入**无 UI Fragment** 实现与 Activity 生命周期的完美绑定，从而杜绝内存泄漏和流量浪费。
- **转场动画的底层匹配逻辑**：学习转场动画效果，通过 `transitionName` 暗号匹配，由系统底层计算坐标和缩放比生成的补间动画欺骗视觉。

## 3. 遇到的疑难杂症及解决方案
1. **转场动画的白块与闪烁断层 (Flicker & Jump)**
   - **问题**：网速稍慢时点击卡片，原位置出现白坑，且详情页出现闪烁和突然跳图的现象。
   - **分析**：Glide 加载大图是异步的，系统底层转场动画抓取到了空相框，导致动画错位。
   - **解决**：在 DetailActivity 中引入 `supportPostponeEnterTransition()` 冻结系统转场，等待 Glide 的 `onResourceReady` 回调后再执行 `supportStartPostponedEnterTransition()` 放行。
2. **由于网络阻塞导致的页面假死假象**
   - **问题**：引入冻结转场后，点击未缓存的图片，等待网络请求期间应用仿佛卡死无响应。
   - **解决**：引入了 **“超时兜底机制 (Timeout Fallback)”**。通过 `postDelayed` 给定 100 毫秒的最长等待阈值，若超时则强行打破冻结状态先弹页面，保证了用户的瞬间响应体验。
