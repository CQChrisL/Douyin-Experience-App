## 2. 学习路径与核心思考逻辑
在整个学习过程中，主要通过**Android 客户端开发与 Java 后端架构的知识映射**来理解 Android 的知识体系。以下是我提炼的核心认知：
- **工程构建体系**：深刻理解了 `build.gradle` 的作用，将其等价于后端的 Maven (`pom.xml`) 依赖管理与构建过程。
- **UI 渲染与操控**：理解了 XML 布局如同 HTML 模板，而 `findViewById` 就是在寻找内存中对象句柄的过程（类似 DOM 树查找）。
- **RecyclerView 的本质**：它并非简单的列表，而是自带**“视图对象连接池”**的渲染引擎，能将成千上万的数据映射到仅有的几个内存 View 对象上。
- **三级缓存架构**：将其与后端的防穿透架构完美对标。通过划拨 1/8 最大可用内存作为 `LruCache`（充当 Guava/Caffeine 本地缓存），结合 Android 内部文件存储机制（充当 Redis 集中式缓存），实现了“内存->磁盘->网络”的优雅降级。

## 3. 遇到的疑难杂症及解决方案
结合我本周的[学习记录]，总结了两个难题：
1. **主线程网络异常 (NetworkOnMainThreadException)**
   - **问题**：在主线程发起同步网络 IO 会导致应用崩溃甚至 ANR（无响应）。
   - **解决**：采用 OkHttp 的 `enqueue` 开启异步子线程执行 IO 任务（类似提交 `Runnable` 到后端线程池），拿到流数据后，严格通过 `Handler(Looper.getMainLooper()).post` 切换回主线程来刷新 UI。
2. **列表图片“残影”与错乱串台**
   - **问题**：在网络有延迟时，快速上下滑动列表，原本滑走的旧图片又会闪烁或者错位显示在新卡片上。
   - **解决**：由于 `ViewHolder` 被系统复用而导致典型并发时序 bug。通过在 `ImageLoader` 中为每次加载引入了 **`ImageView.tag = url` 校验机制**。在异步网络流返回时，核对 ImageView 绑定的 Tag 是否还是当初发请求的 URL，从而完美规避了过期回调造成的脏数据渲染。
