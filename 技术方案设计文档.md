# 抖音经验栏双列图文流技术方案设计文档

## 1. 项目概述

本项目是一个参考抖音经验栏的 Android 双列图文流 Demo，核心目标是实现多栏目图文瀑布流、远端数据拉取、图片缓存加载、详情页预览和共享元素转场。

当前最终版本的主要能力包括：

1. 使用 `TabLayout + ViewPager2 + Fragment` 实现“推荐 / 经验 / 同城”三栏结构。
2. 每个栏目内部使用 `RecyclerView + StaggeredGridLayoutManager` 实现双列图文流。
3. 使用 `OkHttp` 拉取 `picsum.photos` 的真实 JSON 列表数据。
4. 使用 `Glide` 加载列表缩略图和详情页高清图。
5. 通过 `AppGlideModule` 将 Glide 图片请求接入统一的 OkHttp 网络底座。
6. 对列表图、详情图使用不同的缓存策略、优先级和加载时机。
7. 详情页转场和高清主图加载彻底解耦，避免远端图片慢导致转场卡死。
8. 完成相应 A/B 测试和性能调参，最终确定版本提升Demo整体体验。

本项目最终的优化重点不是单纯追求“所有请求最快”，而是围绕图文流场景做资源调度：列表快出缩略图，详情页优先保证转场流畅，高清图可以在转场完成后再加载。

## 2. 整体架构

项目采用单宿主 Activity + 多 Fragment 的结构：

```text
MainActivity
  ├─ TabLayout
  ├─ ViewPager2
  │   ├─ FeedFragment("推荐")
  │   ├─ FeedFragment("经验")
  │   └─ FeedFragment("同城")
  └─ splashOverlay

FeedFragment
  └─ SwipeRefreshLayout
      └─ RecyclerView
          └─ FeedAdapter

DetailActivity
  ├─ ImageView: 详情图片
  ├─ TextView: 标题
  └─ TextView: 点赞按钮
```

核心职责划分：

- `MainActivity`：负责首页容器、Tab 与 ViewPager2 联动、创建遮罩以播放启动动画。
- `FeedFragment`：负责栏目参数、OkHttp JSON 请求、列表数据刷新。
- `FeedAdapter`：负责卡片绑定、列表缩略图加载、点击跳转和转场条件判断。
- `FeedItem`：数据模型，负责定义卡片包含元素的数据类型。
- `DetailActivity`：负责详情页的展示，具体实现包含转场期缩略图占位、转场放行、高清主图加载。
- `NetworkManager`：负责全局 OkHttpClient、连接池和并发调度。
- `MyAppGlideModule`：负责将 Glide 的网络请求替换为 OkHttp 实现。
- `PerformanceConfig`：集中保存转场等待上限和性能日志 tag。

## 3. 数据模型与数据流

 `FeedItem` 数据模型包含：唯一标识符id，标题描述以及图片缩略图和详情图：

```kotlin
data class FeedItem(
    val id: Int,
    val title: String,
    val thumbnailUrl: String,
    val detailUrl: String
)
```

从而实现针对列表页和详情页的不同图片目标：列表页需要快速、稳定、可复用的 500x500 裁剪图；详情页需要更清晰的大图，实现不同的本地缓存策略和调用。

远端 JSON 数据流如下：

```text
FeedFragment
  -> 根据栏目决定 page 参数
  -> OkHttp 请求 https://picsum.photos/v2/list?page={page}&limit=50
  -> org.json 解析 id / author
  -> 过滤目前存在的异常图片 id: 111, 112, 131
  -> 生成 FeedItem(thumbnailUrl, detailUrl)
  -> Handler 切回主线程
  -> 更新 RecyclerView
```

图片 URL 生成策略：

```text
thumbnailUrl = https://picsum.photos/id/{id}/500/500.jpg
detailUrl    = https://picsum.photos/id/{id}/900/1200.jpg
```

项目中曾经在日志里观察到部分 Picsum 图片存在解码异常，因此对部分id做了过滤，避免已知异常源进入列表。

## 4. OkHttp 网络底座设计

### 4.1 全局共享 OkHttpClient

项目通过 `NetworkManager.sharedClient` 维护唯一的 OkHttpClient：

```kotlin
val dispatcher = Dispatcher().apply {
    maxRequests = 12
    maxRequestsPerHost = 6
}

OkHttpClient.Builder()
    .dispatcher(dispatcher)
    .connectionPool(ConnectionPool(5, 5, TimeUnit.MINUTES))
    .connectTimeout(15, TimeUnit.SECONDS)
    .readTimeout(15, TimeUnit.SECONDS)
    .build()
```

这里的关键设计是：

- `Dispatcher(maxRequests=12, maxRequestsPerHost=6)`：限制同时运行的请求数量，防止列表滚动时图片请求过多导致队列和线程压力失控。
- `ConnectionPool(5, 5min)`：保留最多 5 个空闲连接 5 分钟，复用 TCP/TLS 连接，减少重复握手。
- 全局单例：业务 JSON 请求和 Glide 图片请求都围绕同一个网络底座进行复用，避免每个 Fragment 或模块创建独立连接池。

### 4.2 JSON 请求链路

`FeedFragment` 使用 `NetworkManager.sharedClient.newCall(request).enqueue(...)` 发起异步请求。

具体实现：

- 使用 `enqueue`，网络 I/O 不阻塞主线程。
- 使用 `response.use {}`，确保 `ResponseBody` 被关闭，连接可以归还给 OkHttp 连接池。
- Fragment 销毁时调用 `currentCall?.cancel()`，避免页面已销毁后请求继续回调。
- 回调切回主线程前检查 `isAdded` 和 `view == null`，避免无效 UI 更新。

### 4.3 Glide 图片请求链路

项目通过 `MyAppGlideModule` 将 Glide 的网络加载器替换为 OkHttp：

```kotlin
val glideClient = NetworkManager.sharedClient.newBuilder()
    .connectTimeout(30, TimeUnit.SECONDS)
    .readTimeout(30, TimeUnit.SECONDS)
    .build()

registry.replace(
    GlideUrl::class.java,
    InputStream::class.java,
    OkHttpUrlLoader.Factory(glideClient)
)
```

`newBuilder()` 通过浅拷贝的方式基于全局 client 派生一个图片请求 client。这样做的好处是：

- 图片请求和业务请求共享核心的“重资源”底层TCP连接池 ConnectionPool 与线程调度器 Dispatcher。
- 图片请求可以独立设置更长的超时时间，业务请求后续可以添加拦截器等实现轻量级的配置副本。
- 不会创建完全独立的 OkHttpClient，避免连接池分裂。

## 5. Glide 图片加载策略

### 5.1 列表缩略图策略

列表页使用：

```kotlin
Glide.with(holder.itemView.context)
    .load(currentItem.thumbnailUrl)
    .placeholder(android.R.color.darker_gray)
    .priority(Priority.NORMAL)
    .override(500, 500)
    .diskCacheStrategy(DiskCacheStrategy.RESOURCE)
    .centerCrop()
    .into(holder.ivCover)
```

设计原因：

- `Priority.NORMAL`：列表缩略图不能被长期饿住，否则影响首屏体验和导致共享元素条件变差。
- `override(500, 500)`：明确列表图目标尺寸，减少不必要的大图解码。
- `DiskCacheStrategy.RESOURCE`：只缓存转换后的列表资源，避免 `ALL` 同时缓存原始图和转换图带来的双份磁盘压力。
- `centerCrop()`：匹配列表卡片显示方式。

### 5.2 RecyclerView 回收与请求取消

`FeedAdapter.onViewRecycled()` 中调用：

```kotlin
Glide.with(holder.itemView).clear(holder.ivCover)
```

当 ViewHolder 被回收时，清理 ImageView 上仍在进行的旧图片请求，避免已经滑出屏幕的卡片继续占用网络、解码和内存资源。

### 5.3 详情页两阶段加载策略

详情页加载被拆成两个阶段。

阶段一：转场期只使用缩略图缓存。

```kotlin
Glide.with(this)
    .load(thumbnailUrl)
    .onlyRetrieveFromCache(true)
    .diskCacheStrategy(DiskCacheStrategy.RESOURCE)
    .override(500, 500)
    .centerCrop()
    .into(ivDetailCover)
```

这个阶段不会发起远端请求。它只尝试命中列表页已经加载过的缩略图缓存。如果缓存命中，就用于共享元素动画的目标内容；如果缓存未命中，就显示灰色占位并快速放行动画。

阶段二：转场放行后再加载高清主图。

```kotlin
Glide.with(this)
    .load(detailUrl)
    .priority(Priority.IMMEDIATE)
    .diskCacheStrategy(DiskCacheStrategy.DATA)
    .fitCenter()
    .into(ivDetailCover)
```

这个阶段才真正请求详情高清图。`Priority.IMMEDIATE` 会提高 Glide 任务调度优先级，用于实现详情页主图优先于普通列表图，提升当前视图的加载速度，提升用户体验。

## 6. 详情页转场设计

最终版本的点击策略是：

```text
如果列表缩略图已加载完成:
  -> 使用共享元素转场
  -> 详情页尝试从缓存取缩略图
  -> 最多等待 120ms
  -> 放行动画
  -> 转场后加载高清图

如果列表缩略图还没加载完成:
  -> 自动降级为普通跳转
  -> 详情页立即显示占位
  -> 转场后加载高清图
```

实现中通过 `tag_image_loaded` 判断列表缩略图是否已经 `onResourceReady`：

```text
loaded=true  -> mode=shared
loaded=false -> mode=plain_fallback
```

设计的核心原则是：共享元素转场只在有真实源图时启用；没有源图时，强行做共享元素动画只会放大等待和闪烁问题。

转场等待上限为 120ms 也保证了详情页不会因为远端网络或解码慢而长时间冻结。

## 7. A/B 测试与日志指标

为了定位卡顿来源，项目做过共享元素动画与普通跳转的 A/B 测试。测试时记录了：

- 点击到详情页 `onCreate` 的耗时。
- 点击到详情页首帧 `firstPreDraw` 的耗时。
- 转场放行耗时。
- Glide 图片 ready 来源：`MEMORY_CACHE`、`RESOURCE_DISK_CACHE`、`DATA_DISK_CACHE`、`REMOTE`。

### 7.1 普通跳转基线

普通跳转测试日志：

```text
clickToOnCreate = 24-34ms
clickToFirstPreDraw = 63-89ms
glide ready = 182-248ms
source = DATA_DISK_CACHE
```

结论：Activity 启动和页面首帧本身没有问题，普通跳转非常快。因此卡顿不是 Activity 创建慢，也不是布局首帧慢。

### 7.2 共享元素动画初版问题

共享元素测试中观察到：

```text
glide ready elapsed=4354ms source=REMOTE
glide ready elapsed=4463ms source=REMOTE
glide ready elapsed=6506ms source=REMOTE
```

这些日志说明：当列表缩略图还没加载完成时点击卡片，详情页需要远端拉图。共享元素动画会把“目标图没准备好”的问题放大成明显卡顿。

因此第一轮优化加入了自适应降级：只有 `loaded=true` 才走共享元素，否则普通跳转。

### 7.3 连接泄漏验证

早期日志中出现过 OkHttp 连接泄漏提示：

```text
A connection to https://fastly.picsum.photos/ was leaked.
```

后续修复包括：

- JSON 请求使用 `response.use {}`。
- Fragment 销毁时取消未完成 Call。
- RecyclerView 回收时使用 `Glide.clear()` 取消无效图片请求。

最新日志中连接泄漏计数 connection leaked count = 0 说明 OkHttp 连接已经可以正常归还连接池。

### 7.4 列表优先级调参

曾经尝试将列表缩略图设置为 `Priority.LOW`，并把 OkHttp 单 host 并发压到 4。结果在 Picsum/Fastly 较慢时，列表缩略图容易被饿住，从而导致小图长时间不出现。

最终调参为：

```text
列表图：Priority.NORMAL
详情图：Priority.IMMEDIATE
OkHttp：maxRequests=12, maxRequestsPerHost=6
```

最新体验结果：列表缩略图加载稳定，几乎找不到小图为空的场景，详情页转场也保持流畅。

故学习到不能为了给详情页让路而过度压低列表，否则列表缩略图不足会反过来破坏共享元素转场条件。最终策略是在列表首屏可用和详情页高优先级之间做平衡。

## 8. 当前各场景工作链路

### 8.1 App 启动

```text
MainActivity onCreate
  -> 创建 ViewPager2 和默认 FeedFragment
  -> splashOverlay 覆盖界面
  -> FeedFragment 在后台拉 JSON
  -> RecyclerView 绑定数据
  -> Glide 开始加载首屏缩略图
  -> 2s 后 splashOverlay 淡出
```

这将启动动画时间和首屏数据加载时间重叠，减少用户看到空页面的概率。

### 8.2 切换栏目

```text
用户滑动 ViewPager2
  -> 当前 FeedFragment onResume
  -> 若未加载过，则发起 JSON 请求
  -> 旧 Fragment 如销毁视图，则取消当前 JSON Call
```

这样可以避免三个栏目一开始同时请求数据和图片，降低启动时资源竞争。

### 8.3 列表滚动

```text
RecyclerView 复用 ViewHolder
  -> onBindViewHolder 加载 thumbnailUrl
  -> Glide 使用 NORMAL 优先级请求缩略图
  -> 缩略图 ready 后设置 tag_image_loaded=true
  -> ViewHolder 回收时 Glide.clear() 取消旧请求
```

这个链路保证列表图能及时出现，同时避免滑走卡片继续占用资源。

### 8.4 点击已加载缩略图

```text
loaded=true
  -> 走共享元素转场
  -> DetailActivity 从缓存取 thumbnailUrl
  -> 120ms 内放行动画
  -> 转场后 IMMEDIATE 加载 detailUrl
```

这是体验最优路径，适合图片已经出现在列表上的情况。

### 8.5 点击未加载缩略图

```text
loaded=false
  -> 普通跳转降级
  -> DetailActivity 显示 placeholder
  -> 转场后 IMMEDIATE 加载 detailUrl
```

这个路径避免对灰色占位图强行做共享元素动画，优先保证响应速度。

## 9. 问题复盘

### 9.1 为什么不全是 OkHttp 下载能力差的问题导致缩略图和详情图加载困难

日志证明 OkHttp + Glide 能够成功下载和缓存图片：

```text
thumbnail ready source=RESOURCE_DISK_CACHE
detail image ready source=REMOTE
detail image ready source=MEMORY_CACHE
```

下载能力差带来的影响只是我们的缩略图为默认灰图，详情页面为默认黑图。而真正影响用户体验的是调度策略：列表优先级太低会导致缩略图出不来；详情页如果等待远端主图会导致转场卡顿，这种看不到图而又整个页面卡顿做不了交互的体验才是最大的事故。

### 9.2 避免无限制使用 DiskCacheStrategy.ALL

`ALL` 会缓存原始数据和转换后的资源。它复用能力强，但磁盘和解码压力也更大。最终版本中：

- 列表图使用 `RESOURCE`：缓存 500x500 裁剪结果。
- 详情图使用 `DATA`：缓存高清原始数据，方便后续按详情尺寸解码。

这比所有场景统一 `ALL` 更符合当前业务。

### 9.3 priority 不是绝对插队

`Priority.IMMEDIATE` 能影响 Glide 的任务调度顺序，但不能抢占已经在执行的 HTTP 请求，也不能绕过网络本身的 RTT 和 CDN 响应时间。

项目同时做了三件事以实现网络请求的平衡：

1. 限制 OkHttp 并发，避免列表请求洪峰。
2. 列表使用 `NORMAL`，保证缩略图及时出现。
3. 详情使用 `IMMEDIATE`，让主图请求在新任务中优先调度。

## 10. 后续优化方向

目前项目已满足基本目标。后续如果需要优化，可以考虑：

1. 使用 `ListAdapter + DiffUtil` 替代 `notifyDataSetChanged()`。
3. 使用 Retrofit + Kotlinx Serialization 替代手写 `org.json`。
4. 结合图片宽高比动态设置卡片高度，实现更真实的瀑布流。
5. 使用 Android Studio Profiler ，Frame Metrics 等工具量化掉帧情况实现内存分析继续优化体验。
6. 结合更真实，更丰富的数据源进行测试。
