# 抖音经验栏（双列图文瀑布流）技术方案设计文档

## 1. 项目背景与需求分析
本项目为 Android 客户端工程训练营的实战期末作业。目标是从零构建一个对标抖音 APP 首页“经验栏”的双列图片流列表及配套的图文详情页。
**核心业务诉求**：
- 实现异步网络图片加载（指定使用 `OkHttp`）。
- 实现防重复下载的本地历史缓存机制。
- 保证瀑布流在加载大量图片时的滑动流畅度。

## 2. 整体架构设计
为降低代码耦合度并利于演进，项目整体划分为三大核心层：
1. **视图层 (UI Layer)**：`MainActivity` + `DetailActivity`，基于 `RecyclerView` 提供池化视图渲染。
2. **网络引擎层 (Network Layer)**：复用全局的 `OkHttpClient` 连接池，剥离 I/O 线程与主线程。
3. **缓存服务层 (Cache Layer)**：抽离出专门的单例 `ImageLoader`，承载高可用的多级缓存调度逻辑。

## 3. 核心机制设计与底层原理解析

### 3.1 瀑布流渲染与视图池化机制
- **核心组件**：`RecyclerView` + `StaggeredGridLayoutManager(2, VERTICAL)`。
- **复用原理**：由于前端 DOM 节点过多会引发页面瘫痪，Android 采用了 `ViewHolder` 复用池技术（类似后端的数据库连接池）。当卡片滑出屏幕外时，会被回收到缓存池中。此时系统调用 `FeedAdapter.onBindViewHolder` 将底部即将出现的新数据动态注入到这块旧的内存 View 中，从而确保了即使列表有上万条数据，堆内存占用也始终恒定在个位数 M 级别。

### 3.2 异步网络引擎与跨线程通信
- **连接复用**：在 `ImageLoader` 顶部静态实例化唯一的 `OkHttpClient`，以复用其底层的 TCP 保持连接（Keep-Alive），避免了反复发起 HTTP 三次握手造成的网络极度耗时。
- **线程隔离模型**：
  在 Android 系统中，主线程每 16ms 需要刷新一次 UI（60fps），严禁任何网络 I/O 阻塞主线程。因此，我们使用 OkHttp 的 `enqueue` 异步 API，将网络下载任务发往其内部维护的 **Worker 线程池**。
  下载并解码完成位图（Bitmap）后，利用 Android 核心消息队列机制：`Handler(Looper.getMainLooper()).post {}`，将结果作为 Message 投递回主线程的消息队列进行渲染。

### 3.3 高可用三级图片缓存架构（核心亮点）
为解决滑动复用触发反复下载导致的“网络风暴”及 UI 闪烁，项目自主封装了 `ImageLoader` 组件，实现了一套类似大厂方案的三级缓存降级策略：

1. **L1 层：基于 LRU 算法的内存缓存 (Memory Cache)**
   - 机制：动态获取运行时 App 被分配的最大堆内存，划拨其中的 **1/8** 实例化 `android.util.LruCache`。
   - 表现：它的查询耗时在纳秒级。当 200 张超大图片迅速填满这 1/8 空间时，底层的 **Least Recently Used (最近最少使用)** 算法会自动剔除陈旧的 Bitmap 对象，从根本上杜绝了 OOM 内存溢出。
2. **L2 层：持久化磁盘缓存 (Disk Cache)**
   - 机制：利用系统提供的安全沙盒 `context.cacheDir` 作为落地磁盘。
   - 文件命名：为防止 URL 中特殊的 `?/=` 字符导致系统级 FileNotFound 异常，采用 `MessageDigest` 对 URL 进行 **MD5 单向散列**，作为落地的唯一文件名。
   - 策略：只要命中 L2 磁盘，在完成 UI 渲染的同时，会立刻触发一次**反向写入 L1 内存缓存**的操作，确保下一次查询直接命中极速 L1。
3. **L3 层：网络请求底座 (Network)**
   - 策略：仅在 L1 和 L2 全线击穿时触发。一旦 OkHttp 从服务器返回正确字节流并解码，会立即触发**双写策略**（同时压入 L1 内存和写入本地 `FileOutputStream` L2 文件），形成完整数据闭环。

### 3.4 视图防篡改：“Tag 校验”机制
- **问题**：在快速滑动场景下，旧的 `ImageView` 被回收复用去下载“图片 B”。由于网络延迟，之前为“图片 A”发出的异步请求此刻正好返回。若直接渲染，会导致“旧图覆盖新图”的脏数据严重 bug。
- **解法**：发起请求前执行 `imageView.tag = url`。当异步回调抵达主线程时，再次比对 `if (imageView.tag == url)`。如果不一致，说明此 View 已经被分配给了其他数据对象，直接遗弃该过期结果。

### 3.5 组件间路由与参数投递
- **实现**：采用系统级的 `Intent` 进行页面重定向跳转（`DetailActivity`）。
- **原则**：坚决遵循轻量级传参规则，仅在 `putExtra` 中传递 `imageUrl` 和文案信息。目标页接收到 URL 后，直接向 `ImageLoader` 发起加载指令，依赖底层的三级缓存瞬间从内存/磁盘中将其恢复为大图，实现了零等待的秒开体验。

## 4. 后续演进方向
这套架构目前已经具备了商业基础版的雏形，未来的演进方向可围绕以下点展开：
1. **OOM 极致防御**：增加根据目标 ImageView 的物理尺寸对源图进行 `inSampleSize` 的采样压缩降级。
2. **线程池独立**：目前读取本地 File 的操作为了简单暂时跑在主线程。随着文件增多，L2 的读写也可下沉至一个专门的定长 `ExecutorService` IO 线程池中。
